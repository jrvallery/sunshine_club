# Pipeline Quality Evaluation Roadmap

## Purpose

This roadmap defines the next product and engineering milestones for making the
Sunshine Club pipeline trustworthy enough to scale beyond ad hoc QA samples.

The immediate goal is not to process every file faster. The immediate goal is to
measure whether the pipeline is correct, understand where it fails, and create a
repeatable loop for improving OCR, semantic tagging, review routing, and folder
placement without risking source data.

The pipeline should answer these questions for every run:

- Did extraction produce usable text?
- Did OCR fallback run when local OCR was poor, empty, or gibberish?
- Were embeddings generated by the configured provider?
- Did tag assignment use taxonomy definitions, extracted content, and similar
  labeled examples?
- Is confidence calibrated against observed correctness?
- Which files need human review and why?
- What models were called, how long did they take, and what did they cost?

## Current State

The project now has the major pieces needed for an end-to-end quality loop:

- A LangGraph-oriented sample pipeline that can classify, extract, chunk, embed,
  tag, and persist run artifacts.
- Local OCR plus OpenAI OCR fallback for poor, empty, or suspect text extraction.
- Gemini embeddings support, with provider status surfaced in pipeline output.
- Structured LLM tagging support using external or OpenAI-compatible local model
  endpoints.
- A FastAPI and dashboard app for browsing runs, reports, review records, files,
  and artifacts.
- Run deletion support for removing test-run database rows and generated
  artifacts without modifying original source files.
- Model usage reporting groundwork for tracking external calls, runtime, tokens,
  and cost.
- Initial taxonomy and semantic tagging design docs.

Known gaps remain:

- Tagging quality is not yet measured against a durable golden-label set.
- Some confidence scores still represent internal agreement more than real-world
  correctness.
- Embedding failures can degrade decisions unless the run report clearly treats
  them as quality failures.
- Deterministic keyword signals can over-weight incidental terms such as `Tea`.
- OCR confidence alone is insufficient; text length, gibberish detection, page
  coverage, and content plausibility also matter.
- Review records need stronger run lineage so a reviewer can see exactly which
  run, settings, OCR method, model calls, and artifacts produced a decision.

## Non-Goals

These milestones do not include:

- Modifying original corpus files.
- Automatically moving files into final folders.
- Building a full production Google Drive writeback flow.
- Replacing the dashboard frontend wholesale.
- Treating any LLM result as ground truth without evaluation.

## Target Architecture

```text
source file
  -> load file context
  -> classify content type
  -> plan extraction
  -> extract content
  -> validate extracted text
  -> OCR fallback gate
  -> chunk content
  -> embed chunks
  -> retrieve labeled examples
  -> structured LLM tag adjudication
  -> confidence calibration
  -> placement proposal
  -> review routing
  -> persist run artifacts
  -> dashboard review and golden-label update
  -> evaluation report
```

The core architectural rule is that every confidence-producing step must leave
evidence. A result is only useful if the dashboard can show why it happened.

## Milestone 1: Clean Baseline Runs

### Goal

Create a clean operating baseline so future improvements can be compared against
known runs instead of mixed test artifacts.

### Implementation

- Use the delete-run feature to remove throwaway runs from the dashboard.
- Preserve or create one named baseline run using current settings.
- Record run settings with each run:
  - input batch
  - OCR mode
  - OCR fallback mode
  - embedding provider and model
  - LLM tagging provider and model
  - taxonomy version
  - git commit
- Ensure review rows and report pages expose `run_id`.
- Add links from review rows to the producing run report.

### Success Criteria

- The dashboard can distinguish baseline, test, and evaluation runs.
- Each review row shows the run that produced it.
- Clicking a run from review opens the run report.
- Deleting a run removes associated review rows, tag rows, generated artifacts,
  and report records.
- Source files remain read-only and untouched.

## Milestone 2: Golden QA Set

### Goal

Create a human-labeled sample set that represents the real corpus and becomes
the source of truth for measuring progress.

### Implementation

- Select 75 to 150 files across high-risk and common categories:
  - meeting minutes and agendas
  - finance and treasurer records
  - membership rosters and yearbooks
  - tea event records
  - scrapbook pages
  - newspaper clippings
  - historical summaries
  - photos with weak metadata
  - scanned handwritten PDFs
  - image-only PDFs
  - spreadsheets, Publisher files, videos, and technical deferrals
- Store labels in a durable table or JSONL artifact.
- Required label fields:
  - `source_path`
  - `relative_path`
  - `content_class`
  - `correct_primary_tag`
  - `correct_secondary_tags`
  - `ocr_quality_label`
  - `expected_review_required`
  - `sensitive_record`
  - `reviewer`
  - `reviewed_at`
  - `notes`
- Allow labels to be exported to CSV/JSONL for audit and backup.

### Success Criteria

- At least 75 files are labeled before the first real evaluation run.
- Every primary taxonomy family has at least one labeled example, or the gap is
  explicitly documented.
- High-risk categories have multiple labeled examples.
- Labels can be viewed and edited from the dashboard.
- Labels can be exported without reading source files again.

## Milestone 3: Evaluation Runner

### Goal

Create a repeatable command and dashboard flow that runs the current pipeline
against the golden QA set and reports objective results.

### Implementation

- Add a pipeline evaluation command, for example:

```bash
.venv/bin/python -m sunshine_extraction.evaluate_pipeline \
  --golden-labels .local/sunshine-review.sqlite \
  --output-dir /mnt/sunshine/_manifest/sunshine-club-inventory-2026-05-25/eval-runs
```

- Evaluate:
  - content class accuracy
  - primary tag accuracy
  - secondary tag precision and recall
  - OCR acceptable rate
  - OCR fallback rate
  - review routing precision
  - review routing recall
  - embedding provider success rate
  - LLM structured-output validity
  - model calls, runtime, tokens, and cost
- Emit artifacts:
  - `eval-summary.json`
  - `eval-results.jsonl`
  - `eval-confusion-matrix.json`
  - `eval-failures.jsonl`
  - `eval-model-usage.jsonl`

### Success Criteria

- One command runs the golden QA set end to end.
- Every prediction is compared to a human label.
- Failures are grouped by cause, not just listed by file.
- Evaluation output can be loaded into the dashboard.
- The same golden set can be rerun after code changes to compare results.

## Milestone 4: OCR Quality Gate

### Goal

Prevent poor OCR or misleading PDF text extraction from silently flowing into
tagging and embeddings.

### Implementation

- Keep local OCR as the first pass for cost control and privacy.
- Add or harden a `validate_text_extraction` node that checks:
  - text length
  - page coverage
  - OCR confidence
  - gibberish ratio
  - repeated symbol noise
  - table distortion indicators
  - handwritten-text likelihood
  - mismatch between file type and extracted text quality
- Escalate to fallback OCR when:
  - text is empty
  - text is very short for a document-like file
  - extracted PDF text is mostly binary/encoding noise
  - local OCR confidence is low
  - local OCR confidence is high but text is implausibly short
  - local OCR output is gibberish
- Store fallback notes and page-level failures.

### Success Criteria

- Known bad examples such as image-only guest list PDFs do not pass as clean
  text extraction.
- High OCR confidence with tiny text is treated as suspicious.
- Fallback OCR calls are counted in model usage.
- The dashboard shows original OCR snippet, fallback OCR snippet, fallback
  reason, and final selected text.
- Poor extraction routes to review instead of confident tagging.

## Milestone 5: Embedding Retrieval Layer

### Goal

Use embeddings to ground tag decisions in similar human-labeled examples instead
of relying mainly on keyword matches.

### Implementation

- Build an embedding index from golden labels.
- Embed:
  - normalized extracted text
  - path context
  - reviewer notes
  - accepted primary and secondary tags
- Add a `retrieve_labeled_examples` graph node.
- Retrieve top examples before LLM tag adjudication.
- Fail closed when embeddings are expected but unavailable:
  - no silent placeholder fallback in evaluation runs
  - provider failures become run warnings
  - affected files route to review or degraded-confidence mode

### Success Criteria

- Evaluation reports show embedding provider, model, dimensions, and success
  count.
- Retrieved examples are visible in the run report.
- At least 80% of golden-set files retrieve a same-family example in the top 5
  once enough labels exist.
- Placeholder embeddings are never counted as successful model calls.
- Tag failures identify whether retrieval was missing, weak, or misleading.

## Milestone 6: Structured Tag Adjudicator

### Goal

Replace brittle primary-tag assignment with a structured semantic decision that
uses taxonomy definitions, extracted content, deterministic signals, and
retrieved examples.

### Implementation

- Prompt the model with:
  - primary tag definitions
  - allowed secondary tags
  - content class
  - extraction quality
  - source path
  - selected extracted text
  - deterministic candidates
  - retrieved labeled examples
- Require structured JSON output:
  - `primary_tag`
  - `secondary_tags`
  - `confidence`
  - `evidence`
  - `competing_tags`
  - `needs_review`
  - `review_reason`
- Validate output against the taxonomy.
- Reject or retry invalid tags.
- Count local and external model calls separately.

### Success Criteria

- No invalid primary tags are persisted.
- Every tag decision includes evidence.
- The model can explain why incidental terms did not control the decision.
- LLM failures do not crash the run; they route to review or degraded mode.
- Model usage reports include local model calls, external model calls, runtime,
  tokens when available, and external cost when known.

## Milestone 7: Confidence Calibration and Review Routing

### Goal

Make confidence mean "likely correct" instead of "the pipeline sounded certain."

### Implementation

- Compute calibrated confidence from:
  - LLM confidence
  - agreement with retrieved examples
  - deterministic guardrail agreement
  - extraction quality
  - content class certainty
  - sensitive-record status
  - competing tag strength
- Route to review when:
  - extraction is poor
  - OCR fallback failed
  - embeddings failed
  - LLM output is invalid
  - top tags conflict
  - sensitive records have medium confidence
  - placement would be ambiguous

### Success Criteria

- High-confidence predictions have high measured accuracy on the golden set.
- Medium-confidence predictions clearly explain uncertainty.
- Low-confidence predictions route to review.
- Review-required rate is visible by run and by category.
- Evaluation reports include false accepts and false reviews.

## Milestone 8: Placement Rules by Tag

### Goal

Turn accepted semantic tags into folder placement proposals without writing to
the source corpus.

### Implementation

- Use the taxonomy folder table as a routing map:
  - `flat`
  - `by_year`
  - `by_year_month`
- Add a placement proposal node after calibrated tag assignment.
- Extract year hints from:
  - filename
  - source path
  - extracted text
  - document metadata
  - event-specific context
- Persist:
  - proposed folder
  - proposed year or year-month
  - placement confidence
  - evidence
  - reason for review if ambiguous

### Success Criteria

- Placement is generated only after tag confidence is acceptable.
- Ambiguous or missing year routes to review for `by_year` folders.
- The dashboard shows proposed destination without moving files.
- Evaluation reports include placement accuracy for labeled examples with known
  expected destinations.

## Milestone 9: Dashboard Review Loop

### Goal

Make review work efficient enough to support repeated training and evaluation
cycles.

### Implementation

- Review rows should show:
  - run id and run name
  - content class
  - proposed primary tag
  - proposed secondary tags
  - calibrated confidence
  - extraction quality
  - OCR snippet
  - fallback snippet when used
  - retrieved examples
  - model usage summary
  - review reason
- Add filters for:
  - run
  - category
  - extraction quality
  - fallback used
  - model provider
  - needs review
  - confidence bucket
- Allow reviewer actions:
  - accept
  - change tag
  - change content class
  - mark OCR poor
  - defer technical
  - add notes
  - promote to golden label

### Success Criteria

- A reviewer can understand why a file needs review without reading raw JSON.
- A reviewer can jump from a review row to the run report.
- Accepted corrections can become golden labels.
- The review queue can be filtered to one run or one failure mode.
- Dashboard review actions never mutate original corpus files.

## Milestone 10: Production Readiness Gate

### Goal

Define when the pipeline is good enough to process larger batches with customer
confidence.

### Implementation

- Establish acceptance thresholds before running large batches.
- Suggested initial thresholds:
  - content class accuracy: at least 95%
  - primary tag accuracy: at least 88% overall
  - no high-risk category below 80% primary tag accuracy
  - OCR acceptable rate for machine-printed documents: at least 90%
  - external model calls tracked: 100%
  - source file mutation: 0
  - untracked embedding fallback in eval runs: 0
  - false accept rate for sensitive records: 0 known cases in golden eval
- Run a larger representative batch only after the golden set passes.
- Preserve every production run report.

### Success Criteria

- The team can say which categories are reliable and which are not.
- The dashboard clearly separates accepted, review-required, failed, and
  deferred files.
- Production runs have complete lineage from source file to final proposal.
- Customer-facing claims are backed by run reports and evaluation metrics.

## Immediate Next Step

Build Milestone 3 first: the golden-label evaluation runner.

That gives the project a scoreboard. Once there is a scoreboard, OCR fallback,
embedding retrieval, LLM tagging, and confidence calibration can be improved
against the same fixed sample instead of relying on isolated examples.

The recommended first implementation slice is:

1. Add or confirm a `golden_labels` export from the dashboard.
2. Add `sunshine_extraction.evaluate_pipeline`.
3. Run the current pipeline against 20 to 30 existing reviewed files.
4. Emit `eval-summary.json` and `eval-results.jsonl`.
5. Display the evaluation summary in the run report.

This creates the foundation for every later milestone.
