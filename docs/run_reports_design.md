# Run Reports Design

Last updated: 2026-05-27

## Executive Summary

Replace the standalone `Reports` page with per-run reports launched from the
`Runs` page. A run report should be the place to understand one pipeline run:
what is happening now, what happened when it finished, what files changed, what
needs review, what model calls were made, and whether the run improved the
pipeline.

The current dashboard separates execution (`/runs`) from reporting
(`/reports`). That is confusing because most report data only makes sense in the
context of a specific run. The new workflow should be:

```text
Runs -> click run -> Run Report
```

The run report must work for both active and completed runs. While a run is
active, it shows live progress, live log events, partial artifact counts, and
model usage accumulated so far. After completion, the same page becomes the
durable audit report for that run.

## Goals

### Product Goals

- Make every run inspectable from one place.
- Remove ambiguity around what the old `Reports` page means.
- Show meaningful progress while a run is active.
- Show live logs without opening terminal output.
- Show model usage per run:
  - provider
  - model
  - call count
  - input tokens
  - output tokens
  - total tokens
  - model runtime
  - cost when an external paid model is called
  - failures and retries
- Show the same health information currently spread across Reports:
  - route status distribution
  - tag distribution
  - OCR quality
  - placement/privacy health
  - semantic evaluation
  - run-to-run comparison
  - review queue generated by the run
- Preserve auditability: no source files are moved or mutated by reporting.

### Engineering Goals

- Keep the existing `Next.js + FastAPI + SQLite + LangGraph` architecture.
- Extend the existing run tables instead of creating a separate reporting
  system.
- Store normalized model usage records as first-class run artifacts.
- Make partial progress visible from artifacts that exist before completion.
- Use TanStack Query polling for active runs and normal cache refresh for
  completed runs.
- Keep run report APIs deterministic and testable.

### Non-Goals

- Do not build general BI dashboards.
- Do not create a separate analytics database in this milestone.
- Do not estimate cost for local/free models as if they were paid APIs.
- Do not require token/cost accounting to block run completion if a provider
  omits usage fields.
- Do not remove raw JSON artifacts; the report is an interface over them.

## Target Navigation

### Remove Dedicated Reports Navigation

Remove `Reports` as a primary sidebar item.

The old `/reports` route can either:

- redirect to `/runs`, or
- show a short migration page that says reports now live on each run.

Recommended: keep `/reports` as a compatibility redirect to `/runs` during this
milestone, then delete the route later when the workflow is stable.

### Runs Page Behavior

`/runs` remains the run launcher and run history page.

Clicking a run row should navigate to:

```text
/runs/{run_id}/report
```

Open the report in the same browser tab by default. Use normal links so users
can command-click/open in a new tab if they want.

The current right-side run drawer should be demoted or removed. It can remain
temporarily for quick status inspection, but the report page becomes the
canonical run inspection surface.

## Run Report Page

Route:

```text
apps/dashboard/app/runs/[runId]/report/page.tsx
```

API-backed data:

```text
GET /admin/runs/{run_id}
GET /admin/runs/{run_id}/progress
GET /admin/runs/{run_id}/events
GET /admin/runs/{run_id}/results
GET /admin/runs/{run_id}/compare-previous
GET /admin/runs/{run_id}/report
GET /admin/runs/{run_id}/model-usage
```

The page should poll while the run is `queued` or `running`, then stop polling
when it reaches `succeeded`, `failed`, or `cancelled`.

### Header

Show:

- run key
- preset
- status badge
- input root
- output dir
- started time
- completed time
- duration
- command summary
- action buttons:
  - Cancel, if running
  - Import Results, if completed and not imported
  - Rerun
  - Rerun Failed, if failed
  - Open Output JSON

### Status / Progress Band

Only show the progress bar while the run is `queued` or `running`.

Progress data should prefer, in order:

1. explicit live progress events
2. partial artifact row counts
3. selected sample count from run summary
4. indeterminate spinner when total count is unknown

Fields:

- processed files
- total files, when known
- progress percent, when known
- current file, when available
- current node, when available
- elapsed time
- latest log line

Completed runs should show a compact final status band instead of an active
progress bar.

### Key Metrics

Show cards:

- files selected
- files processed
- route candidates
- review required
- failed
- OCR poor/empty
- tag disagreements
- placement needs review
- model calls
- total model tokens
- estimated external cost

## Tabs

### Overview

Purpose: answer “what happened in this run?”

Show:

- final status
- processed/review/failure counts
- route status distribution
- quality distribution
- top primary tags
- most common warnings
- generated artifact counts
- import status
- summary JSON collapsed by default

### Files

Purpose: inspect individual file outcomes.

Use TanStack Table with filters:

- filename/path
- final class
- primary tag
- secondary tag
- route status
- OCR quality
- placement status
- warning
- model provider used

Columns:

- file
- class
- extraction strategy
- quality
- primary tag
- secondary tags
- confidence
- route status
- placement
- warnings
- model usage summary

Clicking a file row should open the existing file/review preview drawer or link
to the file browser record.

### Review Queue

Purpose: inspect what this run asks humans to review.

Show rows from `sample-review-queue.jsonl` or imported `review_items` linked to
this run.

Actions:

- Open in Review page filtered to this run
- Import run results
- Add selected items to review queue

### OCR

Purpose: inspect OCR health for this run.

Show:

- OCR document quality distribution
- pages OK/failed/empty
- local OCR vs fallback OCR usage
- poor/empty/gibberish examples
- OCR fallback notes
- table of OCR documents and pages

Important: this should show whether local OCR was enough or fallback was used,
and which provider handled fallback.

### Tags

Purpose: inspect tagging behavior.

Show:

- primary tag distribution
- secondary tag distribution
- deterministic vs semantic vs LLM assignment source
- LLM agreement/disagreement with deterministic/semantic candidates
- low-confidence tags
- confusion pairs if a semantic eval exists
- nearest golden labels used as evidence

### Placement

Purpose: inspect destination routing quality.

Show:

- placement status distribution
- default privacy distribution
- missing-date queue
- destination path samples
- folder rule used
- reviewer role/default owner

### Model Usage

Purpose: make model cost and runtime visible.

Show aggregate cards:

- total model calls
- successful calls
- failed calls
- retries
- input tokens
- output tokens
- total tokens
- total runtime seconds
- external paid cost
- local model runtime

Table grouped by provider/model:

| Provider | Model | Purpose | Calls | Failures | Input Tokens | Output Tokens | Runtime | Cost |
|---|---|---|---:|---:|---:|---:|---:|---:|
| cortex | gemma4-26b | tag_inspection | 36 | 0 | 120k | 8k | 92s | local |
| cortex | Qwen/Qwen3-Embedding-0.6B | embeddings | 180 | 0 | n/a | n/a | 34s | local |
| openai | gpt-4.1-mini | ocr_fallback | 8 | 1 | 30k | 12k | 55s | $x.xx |

Cost behavior:

- Cortex/local calls: cost should display `local` or `$0 tracked external`.
- OpenAI/Gemini/external paid providers: cost is calculated when pricing is
  configured.
- Unknown pricing: show `unknown`, not `$0`.
- Store raw token/runtime data even if cost cannot be calculated.

### Logs

Purpose: replace terminal watching.

Show:

- live event stream while active
- timestamp
- level
- node
- relative path/current file
- message
- parsed progress payload

Use visual severity:

- info
- warning
- error

Errors should stay visible after completion.

### Artifacts

Purpose: expose generated files.

Show:

- artifact file name
- row count
- size
- last modified
- open/download link

Expected artifacts:

- `sample-pipeline-summary.json`
- `sample-pipeline-results.jsonl`
- `sample-review-queue.jsonl`
- `sample-extraction-results.jsonl`
- `sample-ocr-documents.jsonl`
- `sample-ocr-pages.jsonl`
- `sample-chunks.jsonl`
- `sample-embeddings.jsonl`
- `sample-llm-tag-inspections.jsonl`
- `sample-semantic-examples.jsonl`
- `sample-tag-candidates.jsonl`
- `graph-audit-events.jsonl`
- `graph-batch-summary.json`

### Diff

Purpose: answer “did this run change behavior?”

Show:

- previous comparable run
- changed classifications
- changed primary tags
- changed route status
- changed placement status
- new failures
- fixed failures
- new review-required files
- removed review-required files

This uses the existing `compare-previous` endpoint but should be presented as
human-readable tables, not raw JSON first.

## Data Model

Keep SQLite. Extend the current operational store.

### Existing Tables

Use:

- `pipeline_runs`
- `pipeline_run_events`
- `pipeline_results`
- `review_items`
- `golden_labels`
- `file_index`

### New Table: `pipeline_run_model_usage`

```sql
create table if not exists pipeline_run_model_usage (
  id integer primary key autoincrement,
  run_id integer not null,
  source_path text,
  relative_path text,
  node text,
  purpose text not null,
  provider text not null,
  model text not null,
  status text not null,
  started_at text,
  completed_at text,
  runtime_ms integer,
  input_tokens integer,
  output_tokens integer,
  total_tokens integer,
  estimated_cost_usd real,
  cost_basis text,
  request_id text,
  trace_id text,
  error text,
  metadata_json text not null default '{}',
  created_at text not null default (datetime('now')),
  foreign key(run_id) references pipeline_runs(id)
);
```

Indexes:

```sql
create index if not exists idx_model_usage_run_id
  on pipeline_run_model_usage(run_id);

create index if not exists idx_model_usage_provider_model
  on pipeline_run_model_usage(provider, model);

create index if not exists idx_model_usage_source_path
  on pipeline_run_model_usage(source_path);
```

```json
}
```

## Model Usage Instrumentation

Instrument these provider boundaries:

- LLM tag inspector
  - Cortex/OpenAI-compatible chat
  - Gemini generate content
- OCR fallback
  - OpenAI vision fallback
  - Cortex OCR fallback
- embeddings
  - Cortex embeddings
  - Gemini embeddings
  - placeholder embeddings should be recorded as `local_placeholder` only if
    useful, with zero external cost
- rerank/RAG later, when used

### Usage Extraction Rules

OpenAI-compatible responses:

- Read `usage.prompt_tokens`, `usage.completion_tokens`, `usage.total_tokens`
  when present.
- Support alternate names:
  - `input_tokens`
  - `output_tokens`
  - `total_tokens`

Gemini responses:

- Read `usageMetadata.promptTokenCount`
- Read `usageMetadata.candidatesTokenCount`
- Read `usageMetadata.totalTokenCount`

Cortex responses:

- Read OpenAI-compatible `usage` when available.
- Preserve `trace_id` and `x-request-id` when exposed.
- Cost basis is `local` unless the gateway reports an external cost.

OCR responses:

- If provider returns no token usage, still record call count and runtime.
- If OCR uses a paid vision model, estimate from configured pricing when token
  counts are available.

## Cost Configuration

Add an optional pricing file:

```text
config/model_pricing.json
```

Shape:

```json
{
  "openai:gpt-4.1-mini": {
    "input_per_1m_tokens_usd": 0.40,
    "output_per_1m_tokens_usd": 1.60
  },
  "gemini:gemini-2.5-flash": {
    "input_per_1m_tokens_usd": null,
    "output_per_1m_tokens_usd": null
  },
  "cortex:gemma4-26b": {
    "cost_basis": "local"
  }
}
```

Rules:

- Missing pricing means `estimated_cost_usd=null` and `cost_basis=unknown`.
- Local Cortex models use `estimated_cost_usd=null` and `cost_basis=local`.
- Never hard-code a paid provider price directly in pipeline code.

## API Design

### `GET /admin/runs/{run_id}/report`

Returns the aggregate report model:

```json
{
  "run": {},
  "progress": {},
  "overview": {},
  "distributions": {},
  "review_queue": {},
  "ocr": {},
  "tags": {},
  "placement": {},
  "model_usage": {},
  "artifacts": [],
  "diff": {}
}
```

This endpoint can assemble data from:

- run row
- progress endpoint
- summary JSON
- JSONL artifacts
- model usage table/artifact
- compare-previous
- review items imported from this run

### `GET /admin/runs/{run_id}/model-usage`

Returns normalized model usage:

```json
{
  "run_id": 123,
  "summary": {
    "total_calls": 44,
    "failed_calls": 1,
    "input_tokens": 150000,
    "output_tokens": 12000,
    "total_tokens": 162000,
    "runtime_ms": 220000,
    "estimated_external_cost_usd": 1.23
  },
  "by_provider_model": [],
  "rows": []
}
```

### `GET /admin/runs/{run_id}/artifacts`

Returns artifact metadata and links.

### `GET /admin/runs/{run_id}/events`

Keep existing endpoint. Ensure it returns enough live log events for active
runs. Use polling initially.

Optional later improvement: add server-sent events.

## Success Criteria

### Navigation

- There is no primary `Reports` sidebar item.
- Clicking a run from `/runs` opens `/runs/{run_id}/report`.
- `/reports` does not remain a confusing standalone reporting surface.

### Active Runs

- A running run report shows:
  - spinner/progress bar
  - processed/total count when known
  - current status
  - live log events
  - partial artifact counts
  - partial model usage if any calls have completed
- Polling stops automatically after terminal status.

### Completed Runs

- A completed run report shows:
  - final status
  - file counts
  - route/review/failure counts
  - OCR quality
  - tag distribution
  - placement/privacy health
  - review queue generated by the run
  - artifacts
  - diff vs previous comparable run

### Model Usage

- Each external or local model call is represented in model usage data.
- Usage is grouped by provider/model/purpose.
- Token counts are shown when providers return them.
- Runtime is shown for all model calls.
- External cost is shown only when pricing is configured.
- Cortex/local cost is not misrepresented as paid spend.
- Failed model calls are counted and visible.

### Auditability

- Raw run artifacts remain available.
- Report data can be traced back to run ID and artifact paths.
- Missing artifacts produce explicit warnings, not silent empty reports.

### Tests

- API tests cover report endpoint shape.
- API tests cover model usage aggregation and cost behavior.
- API tests cover active-run progress payload.
- Dashboard build passes.
- Existing review/file/run tests still pass.

## Risks And Mitigations

### Risk: Providers Do Not Return Token Usage

Mitigation: record call count, runtime, provider/model, and `usage_status`.
Token and cost fields can be null.

### Risk: Active Run Progress Is Partial

Mitigation: show indeterminate progress when total is unknown. Update as soon as
selected file count or artifact counts become available.

### Risk: Report Endpoint Becomes Too Large

Mitigation: return aggregate data first. Large file tables use paginated
endpoints or artifact-backed virtualized loading.

### Risk: Cost Estimates Become Stale

Mitigation: use config-driven pricing and store `cost_basis`. Do not bake prices
into provider code.

### Risk: Old Reports Page Is Still Useful

Mitigation: move its useful panels into run reports first, then remove the
sidebar link. Keep `/reports` as a redirect for one milestone.

## Recommended Milestone Scope

### Phase 1: Run Report Shell

- Add `/runs/{run_id}/report`.
- Link run rows to report page.
- Move run status, progress, events, results, and diff into the report.
- Remove Reports from sidebar.

### Phase 2: Report Aggregates

- Add `/admin/runs/{run_id}/report`.
- Add artifact listing.
- Move current Reports panels into run-specific tabs.

### Phase 3: Model Usage

- Add model usage table.
- Add `sample-model-usage.jsonl`.
- Instrument LLM tags, OCR fallback, and embeddings.
- Add cost configuration and aggregation.

### Phase 4: Polish And Validation

- Add table filters.
- Add missing artifact warnings.
- Add CSV/JSON export for run report.
- Add tests and dashboard screenshots if Playwright is available.
